## Adding a custom template

To add a new template (e.g., `template.api.rust` for Rust/Axum):

### 1. Create the template directory

```
templates/api/
└── rust/
    ├── template.yaml                    # Template metadata
    ├── .github/
    │   └── workflows/
    │       └── ci.yml.tmpl              # CI workflow template
    ├── .ci-local/
    │   └── ci-local.sh.tmpl
    └── .gitignore.tmpl
```

### 2. Write the template.yaml

```yaml
kind: forge-template
contract_version: 0.1.0
status: published
template:
  id: template.api.rust
  stack: rust
  display_name: Rust API (Axum)
  purpose: Rust/Axum API with clippy + cargo test CI
assets:
  - source: .github/workflows/ci.yml.tmpl
    destination: .github/workflows/ci.yml
    role: CI workflow
  - source: .ci-local/ci-local.sh.tmpl
    destination: .ci-local/ci-local.sh
    role: local CI runner
```

### 3. Register in forge-init.sh

```bash
# In SUPPORTED_TEMPLATES
SUPPORTED_TEMPLATES=(
  ...
  "template.api.rust"
)

# In IMPLEMENTED_TEMPLATES
IMPLEMENTED_TEMPLATES=(
  ...
  "template.api.rust"
)
```

### 4. Add a generation function

```bash
generate_template_api_rust() {
  local project_name="$1"
  local dest="$2"
  local tmpl_root="$REPO_ROOT/templates/api/rust"

  print_step "generating: template.api.rust -> $dest/$project_name"

  mkdir -p "$dest/$project_name/.github/workflows"
  mkdir -p "$dest/$project_name/.ci-local/hooks"

  render_template "$tmpl_root/.github/workflows/ci.yml.tmpl" \
    "$dest/$project_name/.github/workflows/ci.yml" \
    "$project_name"

  # ... other files
}
```

### 5. Add a case in the generation dispatch

```bash
template.api.rust) generate_template_api_rust "$project_name" "$destination" ;;
```

### 6. Test with dry-run

```bash
scripts/forge-init.sh \
  --template template.api.rust \
  --project-name my-crate \
  --destination /tmp/test \
  --dry-run
```

## Extending CI families

The `.ci-local/` family is shared across all templates. To add a new hook:

### 1. Add the hook script

```bash
# templates/shared/ci-local/hooks/pre-merge
#!/usr/bin/env bash
set -euo pipefail
# Run before merging a branch
echo "Running pre-merge checks..."
```

### 2. Update ci-local.sh to reference the new hook

```bash
# In .ci-local/ci-local.sh template
run_hook() {
  local hook="$1"
  local hook_path=".ci-local/hooks/$hook"
  if [[ -x "$hook_path" ]]; then
    "$hook_path"
  fi
}

# Add to the hook chain
run_hook pre-merge
```

### 3. Update install.sh to register the hook

```bash
# In .ci-local/install.sh
ln -sfn "../../.ci-local/hooks/pre-merge" ".git/hooks/pre-merge"
```

## Adding a new generator

To add a custom generator (e.g., `generator.changelog.init`):

### 1. Create the generator directory

```
generators/changelog/
└── init/
    ├── generator.yaml
    └── CHANGELOG.md.tmpl
```

### 2. Write generator.yaml

```yaml
kind: forge-generator
contract_version: 0.1.0
status: published
generator:
  id: generator.changelog.init
  purpose: Add CHANGELOG.md scaffolding to any project
  composable: true
assets:
  - source: CHANGELOG.md.tmpl
    destination: CHANGELOG.md
    role: initial changelog
```

### 3. Register and implement in forge-init.sh

```bash
# In SUPPORTED_GENERATORS
SUPPORTED_GENERATORS=(
  ...
  "generator.changelog.init"
)

# Implementation
generate_changelog_init() {
  local project_name="$1"
  local dest="$2"
  local gen_root="$REPO_ROOT/generators/changelog/init"

  render_template "$gen_root/CHANGELOG.md.tmpl" \
    "$dest/$project_name/CHANGELOG.md" \
    "$project_name"
}
```

## Template variable substitution

All `.tmpl` files support these substitution tokens:

| Token | Replaced with |
|-------|--------------|
| `{{PROJECT_NAME}}` | value of `--project-name` |
| `{{STACK}}` | template stack ID |
| `{{CONTRACT_VERSION}}` | forge contract version |
| `{{GEN_DATE}}` | generation date (ISO 8601) |

Example template:

```yaml
# .github/workflows/ci.yml.tmpl
name: CI — {{PROJECT_NAME}}
on: [push, pull_request]
# Generated by javi-forge {{CONTRACT_VERSION}} on {{GEN_DATE}}
```

## Contributing a template or generator

1. Fork `javi-forge`
2. Add the template/generator following the structure above
3. Run `scripts/forge-init.sh --list-contracts` — your new ID should appear
4. Test with `--dry-run`
5. Open a PR with:
   - The new template/generator directory
   - Updated `forge-init.sh` registration
   - A dry-run output example in the PR description

## Running forge-init.sh in CI

Use javi-forge in a CI pipeline to scaffold consistent project structures:

```yaml
# .github/workflows/scaffold.yml
name: Scaffold project
on:
  workflow_dispatch:
    inputs:
      template:
        description: 'Template ID'
        default: 'template.api.go'
      project_name:
        description: 'Project name'
        required: true

jobs:
  scaffold:
    runs-on: ubuntu-latest
    steps:
      - name: Clone javi-forge
        run: |
          git clone https://github.com/JNZader/javi-forge.git /tmp/javi-forge

      - name: Generate project
        run: |
          /tmp/javi-forge/scripts/forge-init.sh \
            --template "${{ inputs.template }}" \
            --project-name "${{ inputs.project_name }}" \
            --destination "${{ github.workspace }}/generated"

      - name: Upload artifact
        uses: actions/upload-artifact@v4
        with:
          name: generated-${{ inputs.project_name }}
          path: generated/
```

## Forking javi-forge

javi-forge is designed to be forked for custom team templates:

| File to customize | Purpose |
|------------------|---------|
| `templates/<stack>/*/ci.yml.tmpl` | Stack CI workflow |
| `generators/*/generator.yaml` | Generator metadata |
| `scripts/forge-init.sh` | Generation logic and contract IDs |
| `lib/common.sh` | Shared utilities for generated projects |
